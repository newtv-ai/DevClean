# G0 repository, license, CI, and release-evidence protocol

## Boundary

G0 is not a local-test checkbox. It closes only when the project owner approves the license for a
fixed source revision and real GitHub-hosted Windows CI/CodeQL runs for that same revision and wheel
have passed. This protocol records those external facts; it does not publish, tag, upload, or sign
anything.

Start from [the incomplete G0 template](templates/g0-release-manifest.template.json). Keep
`conclusion=INCOMPLETE` until every required evidence file is present and validated.

## Mechanical source observation

From a clean checkout of the exact revision, run:

```powershell
uv run --frozen python scripts/audit_source_boundary.py `
  --source-revision <full-commit-sha> `
  --output <private-evidence-dir>\source-audit.json
```

The auditor hashes the bounded source tree, verifies the current empty runtime dependency/plugin
surface, checks the console entry point, exact `GPL-3.0-or-later` declaration,
`license-files=[LICENSE, THIRD_PARTY_NOTICES.md]`, and rejects vendored path segments associated
with BleachBit/CleanerML/Winapp2/Sifty. For a release-shaped 40/64-character Git object ID it also
requires the source root to be that checkout's top level, `HEAD` to equal the requested revision,
and the worktree (including untracked files) to be clean. The `LICENSE` file must be the canonical
GNU GPLv3 UTF-8/LF text: exactly 35,149 bytes with SHA-256
`3972dc9744f6499f0f9b2dbf76696f2ae7ad8af9b23dde66d6af86c9dfb36986`; the G0 validator
independently requires that digest. Its output deliberately says
`owner_license_decision_proven=false` and `originality_proven=false`; mechanical scanning cannot
make either claim.

## Owner attestation

The project owner creates `owner-attestation.json` with exactly these fields:

```json
{
  "schema_version": "1.0.0",
  "source_revision": "<full-commit-sha>",
  "decision_at": "2026-07-11T00:00:00Z",
  "owner_id": "<non-personal-stable-id>",
  "license_expression": "GPL-3.0-or-later",
  "approved": true,
  "no_third_party_rule_copy_attested": true,
  "notes": "Scope and provenance reviewed for this revision."
}
```

`owner_id` must appear in the manifest's `review.author_ids`. This is authorization evidence and
cannot be generated by Codex, CI, or another model on the owner's behalf.

## Real CI evidence

Push the fixed revision to the real GitHub repository. The checked-in workflow must complete on
Python 3.11, 3.12, and 3.13, with schema validation, dependency audit, tests/coverage, Ruff, Mypy,
the clean-runtime build, reproducibility checks, SBOM validation, and release-payload validation.
Download and retain:

- the wheel/SBOM/checksum artifact and `release-validation.json` produced by
  `scripts/build_release.ps1`;
- pip-audit JSON produced with `--format json` (all `vulns` arrays empty and `fixes=[]`);
- the immutable workflow run/attempt IDs and the exact workflow-file SHA-256;
- the CodeQL run bound to the same commit, language `python`, build mode `none`, query suite
  `security-extended`, and no unresolved error-severity alert.

The CI attestation is a closed JSON contract with source/artifact hashes, three Python job rows,
coverage, all quality flags, the dependency-audit evidence SHA-256, the release-validation evidence
SHA-256, `actions_pinned=true`, `persist_credentials_false=true`, and
`inventory_only_contract_tests_passed=true`. The CodeQL attestation separately binds its run and
workflow hash. Do not use a local simulation or screenshot as a substitute for the real run.
Start from the intentionally failing
[CI](templates/g0-ci-attestation.template.json),
[CodeQL](templates/g0-codeql-attestation.template.json), and
[owner](templates/g0-owner-attestation.template.json) templates; change a field to `PASS`/`true`
only when the corresponding immutable external evidence directly supports it.

For an uncommitted local build, `release-validation.json` uses
`source_revision=WORKTREE_UNCOMMITTED`; the G0 validator rejects that as release evidence. A final
candidate build must pass `-SourceRevision <full-commit-sha>` or run inside the matching Git
checkout.

## Manifest validation

The G0 manifest must reference exactly one artifact of every required kind:

- `PRODUCT_ARTIFACT`
- `OWNER_ATTESTATION_JSON`
- `SOURCE_AUDIT_JSON`
- `CI_ATTESTATION_JSON`
- `CODEQL_ATTESTATION_JSON`
- `DEPENDENCY_AUDIT_JSON`
- `RELEASE_VALIDATION_JSON`

Then run:

```powershell
uv run --frozen python scripts/validate_gate_evidence.py <g0-manifest.json>
```

The validator recomputes every byte count/hash, rejects reparse/hard-link evidence aliases, checks
all closed contracts and cross-bindings, and returns nonzero unless every required check and the
manifest conclusion are `PASS`. Even a valid G0 does not close G1/G2 and does not authorize cleanup.
