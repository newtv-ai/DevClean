# ADR-002：受限 AI 审查包，而非 AI 文件系统控制

- 状态：Accepted
- 日期：2026-07-12
- 决策所有者：Reclaimer maintainers

## 背景

用户要求把系统无法确定、但可能属于开发/AI 缓存的文件交给 Codex 或 Claude 逐项解释；模型仍不确定时才由用户决定。早期 ADR 完全排除 AI，但 GUI 已实现受限的人工中转审查包，因此旧 ADR 已不符合实际产品。

模型无法从路径和元数据证明文件的引用完整性或可重建性；路径还可能泄露项目、软件与使用习惯。模型输出也不能成为文件系统权限、路径选择或任意命令的来源。

## 决策

1. AI 只接收 GUI 中已存在的、最多 50 个开发缓存候选项的脱敏元数据；不发送文件内容，不自动联网，不保存 API Key，不启动本地模型。
2. 审查包包含随机 `review_id` 和固定 `item_id`；模型回复只允许 `DELETE`、`KEEP` 或 `UNSURE` 加逐项解释。
3. 回复必须是严格 JSON，恰好覆盖当前批次全部 ID。额外路径、命令、通配符、未知/重复/遗漏 ID、额外字段、非标准常量都拒绝。
4. `DELETE` 不能扩大候选集合，只能触发本机对原扫描记录的永久删除尝试；本机仍执行保护路径、普通文件、固定卷、句柄身份与最终路径复核。
5. `UNSURE` 和任何本机复核失败都转入用户决定的回收站路径。`KEEP` 不执行删除。
6. 不存在持久 `ai_feedback` 数据库表，也不把模型回复导入旧的 SQLite plan/control 平面；审查批次只在当前 GUI 会话内存在。

## 后果

- AI 可以帮助解释已识别缓存，但不拥有文件系统范围、执行命令或绕过确定性保护的能力。
- 复制到 Codex/Claude 是用户主动的数据披露；路径虽然替换主目录为 `<USER>`，其他目录名仍可能敏感。后续应提供更强的路径最小化和明确的隐私提示。
- 手工粘贴没有可验证的模型来源；它是用户选择的工作流，不应被宣传为“AI 已认证”。未来若接入官方 API，仍必须保留同一批次绑定、本机复核和离线可用的用户决定路径。
- AI `DELETE` 的语义正确性不能由 schema 证明。因此优先方向是为 Hugging Face、Conda、pip、uv、npm/pnpm、Docker、Ollama 和 IDE 缓存建立确定性适配器，减少路径启发式与 AI 判断的比例。

## 被拒绝的方案

- 让模型返回任意绝对路径、目录、glob、PowerShell 或 CLI：会绕过扫描快照与范围约束。
- 将模型输出直接写入 `ai_feedback`/SQLite 并在下次启动自动执行：会把不可信文本变成持久执行授权。
- 把模型 `DELETE` 当作安全证明：模型只能提供解释，最终仍受本机身份复核与硬保护规则限制。
- 自动上传扫描结果或默认接入云端 API：违反最小披露与用户明确发起原则。
