# DevClean 与开源磁盘工具对比审计

审计日期：2026-07-16。本文只使用项目官方仓库或官方文档，并区分“已经实现”“部分实现”“计划中”和“明确不采用”。功能表述不等于对第三方项目安全性的背书。

## 结论

DevClean 不应复刻某一个通用清理器。最合适的组合是：

- 借鉴 BleachBit 的显式预览、清晰选项和可扩展分类思想；
- 借鉴 FluentCleaner/winapp2 的应用覆盖、分类搜索和规则可读性；
- 借鉴 Sifty 的开发者缓存覆盖、默认不执行、批量标记、AI 仅建议、历史与恢复；
- 借鉴 InstallerClean 的“权威厂商 API 优先、降级只能增加保留项”、运行状态阻断和执行后复扫；
- 借鉴 Czkawka/Krokiet 的扫描缓存、可复用核心和多种空间分析；
- 借鉴 WinDirStat 的目录/类型聚合、逻辑与物理大小区分、文件变化监视和可视化导航；
- 借鉴 dupeGuru 的专业重复文件工作流，但不把重复组自动转换成删除授权。

DevClean 的差异化核心是：AI/开发工具覆盖 + 扫描阶段零副作用 + 用户可直接选择 + AI 可选 + 精确文件身份 + 强确认 + SQLite 意图日志 + 私有隔离恢复/永久清除双路径。

## 对比矩阵

| 项目 | 可取长处 | 主要边界/风险 | DevClean 状态 |
|---|---|---|---|
| [BleachBit](https://github.com/bleachbit/bleachbit) / [CleanerML](https://docs.bleachbit.org/cml/cleanerml.html) | Preview 后再 Delete；分类选项；CleanerML/Winapp2 扩展生态 | CleanerML 不只是数据，支持进程、注册表、删除、截断等动作；直接复用会绕开本项目执行边界 | 预览、默认零选择、分类已实现；不调用 `--clean`；外部规则未来只能先进入无执行权预览层 |
| [FluentCleaner](https://github.com/builtbybel/FluentCleaner) | WinUI 3 体验；约数千条 winapp2 规则；分类、搜索、自定义库 | 自定义规则及 PowerShell 能扩大执行面；社区规则质量不等同于本机删除授权 | 统一表格、来源域/类别、搜索筛选已实现；不直接执行 winapp2/PowerShell；数据规则系统列为后续 |
| [Sifty](https://github.com/Vortrix5/sifty) | dry-run 默认；批量标记；开发产物、重复、空间分析；AI 仅建议；审计、恢复；CLI/TUI 分层 | “全部回收站”依赖 Shell/Send2Trash 语义；还包含提权、系统管理等更大范围 | AI 惰性导入、批量计划、开发缓存、重复分析、意图日志、恢复已实现；主进程拒绝提权；不用可能退化为永久删除的 Shell 桥 |
| [InstallerClean](https://github.com/no-faff/InstallerClean) | Windows Installer API 判定；不完整降级只增加“仍需要”；事务/重启状态阻断；执行后复扫；明确部分成功 | 专门针对 `C:\Windows\Installer`，需要管理员权限，不适合作为通用路径启发式 | “降级保护单调性”和执行后核验已采纳；MSI 专用适配器未实现，在此之前该系统目录只报告 |
| [Czkawka/Krokiet](https://github.com/qarmin/czkawka) | 后续扫描缓存；重复/大文件/临时/空项等分析；核心与前端分离 | 通用空间分析不能证明某文件可安全删除 | 同会话变更监视增量、重复/大文件和核心/UI 分层已实现；跨重启持久增量未实现 |
| [WinDirStat](https://github.com/windirstat/windirstat) | 目录树、扩展名/类型、treemap；逻辑/物理大小；文件监视；CSV；丰富导航 | 用户动作和自定义命令范围较大；可视化不构成清理证据 | 目录/类别汇总、逻辑/已分配大小、搜索、变化监视已实现；treemap、扫描保存/载入未实现 |
| [dupeGuru](https://github.com/arsenetar/dupeguru) | 成熟的重复文件专用 GUI 与核心分层 | 重复不等于副本；正本选择和参考目录需要独立策略 | 有界 SHA-256 重复分析已实现；不会自动指定正本或生成删除选择 |
| [temp_cleaner](https://github.com/isubroto/temp_cleaner) | 简单、低学习成本 | 仓库规模和公开工程证据很有限，不足以作为执行安全基础 | 仅借鉴“简单进度与清晰操作”；不作为架构基础 |

## 已吸收的改进

1. 扫描、分类、AI 导入不执行删除；每次扫描完成默认零选择。
2. 用户可以直接选择所有本地执行策略允许的文件；AI 是可选解释器，不是权限门槛。
3. 浏览器、VS Code、缩略图等已知缓存与真正的系统 `REPORT_ONLY` 拆分，不再一并锁死。
4. 一次用户计划最多 256 个精确文件；底层仍按每批 32 个写独立意图日志。最终页一次列出全部路径、大小、计划摘要和清理方式。
5. 永久清除先移入同卷私有隔离区，持久化 `PURGE_PENDING` 后再按已验证句柄处置；可恢复模式保留隔离对象。
6. 隔离目录在批准根下以随机批次名原子创建，DACL 从创建瞬间生效；已存在路径绝不接管。
7. 目标文件句柄不共享写入或删除；最终路径、128 位文件 ID、时间戳、大小、属性、单硬链接等全部复验。
8. 同一 GUI 会话使用 `ReadDirectoryChangesW` 作为失效提示；任何不确定性回退全量扫描。
9. 分类按来源域、具体类别、队列、执行上限、风险、证据、恢复能力展示，并提供目录/类别汇总和重复文件页。
10. 开源仓库补齐贡献指南、行为规范、变更日志、问题模板、PR 模板、安全报告入口和 CI/CodeQL。

## 明确不采用

- 不把 CleanerML、winapp2、脚本或模型输出直接变成执行指令。
- 不在扫描过程中自动清理，也不提供默认勾选。
- 不声称“可重新下载”等于“可恢复”；私有或自建产物可能无法重建。
- 不对 Windows Update、WinSxS、`Windows.old`、`C:\Windows\Installer` 做路径猜测删除。
- 不使用可能在回收站不可用时退化为永久删除的 Shell 删除桥。
- 不把目录大小、文件年龄、扩展名、AI 建议或重复哈希单独当成删除授权。

## 下一阶段差距（非当前完成项）

| 优先级 | 差距 | 完成标准 |
|---|---|---|
| P1 | 跨重启增量 | 持久索引 + NTFS USN 校验；卷/根身份、journal ID、USN 范围任一不可信即全扫 |
| P1 | 规则覆盖 | 仅数据分类 schema、版本/来源/签名/测试；第三方规则默认 preview-only，不能携带命令或执行权限 |
| P1 | 大结果浏览 | 分页/虚拟列表，不因每队列 500 项展示上限隐藏可操作候选 |
| P2 | 可视化 | treemap 或等价交互图，并继续区分逻辑、已分配和实际释放空间 |
| P2 | 厂商适配器 | 每个适配器有权威 API、版本锁、运行状态阻断、预览与执行后复扫；降级只能增加保留项 |
| P2 | AI 跨会话 | 本地持久、过期、签名绑定的复核会话；重启后仍不能扩大候选或授权执行 |
